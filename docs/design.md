# AI 模拟面试 Agent — 设计文档

研究/设计日期：2026-06-11

## 这页解决什么问题

设计一个作品集级别的 AI 应用：输入**岗位描述（JD）+ 目标公司 + 个人简历**，由一个 AI 面试官进行**回合制语音模拟面试**，过程中能针对性出题、实时追问挤水、按结构化流程推进，结束后生成评估报告。

定位：**技术练手 + 作品集项目**，目标岗位是「AI 应用工程师 / 大模型应用全栈工程师」（依据：个人知识库《AI应用工程师招聘市场调研 2026-06-09》）。设计取舍以「能讲出工程深度、命中 JD 关键词」为第一优先级。

## 为什么是这个项目

调研里 AI 应用工程师的核心能力点，本项目几乎全覆盖：

| JD 要求的能力 | 本项目如何体现 |
|---|---|
| Agent 工作流（任务拆解、状态、上下文管理） | 面试本身是一个多阶段状态机 Agent |
| Prompt 工程 / 结构化输出 | 针对性出题、答案评分、评估报告全是结构化 LLM 调用 |
| 上下文工程 | JD + 公司 + 简历 解析成结构化候选人画像，贯穿全程 |
| 流式输出 / SSE | 提问、报告生成走流式 |
| 工程化交付 | Docker、可观测（Langfuse）、评测闭环 |
| 前端体验 | 语音对话工作台、转写流、报告可视化 |

「面试」场景天然能展开讲 Agent 工程深度：防止追问无限循环、判断「挤水」该不该继续深挖、评分 rubric 设计——这些比通用聊天机器人更有说服力。

## 已确认的关键决策

1. **核心用 LangGraph 状态机**，而非单一大 prompt（为工程含金量，不为省事）。
2. **交互形态：回合制语音**（你说完 → STT → LLM → TTS 播报）。架构上把 STT/LLM/TTS 解耦，**预留实时流式（Realtime）升级**，升级时只换语音实现、不动 Agent。
3. **面试智能度全要**：针对性出题 + 实时追问挤水 + 结构化流程 + 评估报告。
4. **语音 STT/TTS：MVP 默认用全免费开源本地方案**——STT 用 `faster-whisper`（本地推理），TTS 用 `edge-tts`（免费）。零语音成本。底层模型/服务都藏在 `STT`/`TTS` 抽象接口之后，效果不满意时可平滑切换到云 API（阿里云 DashScope Paraformer + CosyVoice，或讯飞开放平台，均有免费额度），只改实现类、不动主体架构。
5. **MVP 推迟**：实时流式语音、公司信息联网调研 RAG、多用户/账号、数字人/视频。

## 架构总览

```
输入(JD / 公司 / 简历)
   → intake 解析层 → InterviewContext(候选人画像 + 公司侧重 + 面试计划)
   → interviewer 面试状态机(LangGraph)：提问 → 作答(STT) → 评估打分 → 路由判定 ↺
   → evaluator 评估报告
前端 web 工作台贯穿全程（语音录制、SSE 流式、阶段进度、报告可视化）
voice 适配层为状态机提供 STT/TTS（可插拔）
```

## 模块设计

每个模块单一职责、通过明确接口通信、可独立理解与测试。

### 1. 输入解析层 `intake`

- **做什么**：把原始输入转成结构化的 `InterviewContext`。
  - 简历解析：PDF / DOCX / Markdown → 候选人画像（经历、项目、技能、亮点、可深挖点）
  - JD + 公司文本 → 岗位要求、技术栈、公司侧重点
- **怎么用**：`build_context(resume_file, jd_text, company_text) -> InterviewContext`
- **依赖**：文档解析库（如 `unstructured` / `pypdf` / `python-docx`）+ 一次 Claude 结构化抽取调用（结构化输出 / JSON schema）。
- **边界**：只负责「原料 → 结构化上下文」，不涉及出题与对话。

### 2. 面试计划生成 `planner`

- **做什么**：基于 `InterviewContext` 生成 `InterviewPlan`——每个阶段的考察目标 + 候选问题池。问题必须 ground 在简历项目与 JD 要求上（这是「针对性出题」的核心）。
- **怎么用**：`make_plan(context) -> InterviewPlan`
- **阶段**：自我介绍 → 技术基础 → 项目深挖 → 行为面试 → 候选人反问 → 收尾。
- **依赖**：一次 Claude 调用，结构化输出。

### 3. 面试状态机 `interviewer`（LangGraph，项目核心）

- **做什么**：驱动整场面试的多轮对话与决策。
- **节点**：
  - `ask`：根据当前阶段与计划生成下一个问题（首问或追问）
  - `answer`：接收候选人语音转写文本（由 voice 层提供）
  - `evaluate`：对该回答按 rubric 打分（完整度、深度、相关性、是否有水分）
  - `route`：条件边，依据评分与状态决定下一步
- **路由判定（条件边）**：`追问挤水 / 进入下一题 / 切换阶段 / 结束面试`
- **状态 `InterviewState`**：当前阶段、已问问题列表、**追问预算**（每题/每阶段上限，防无限循环）、累计评分记录、对话历史。
- **怎么用**：编译成 LangGraph graph，按回合 step 推进；状态可持久化（checkpoint），支持中断与回放。
- **面试可讲点**：追问预算与终止条件、阶段目标达成判定、状态持久化、条件路由设计。

### 4. 语音适配层 `voice`（可插拔）

- **做什么**：提供语音输入输出能力。
- **接口**：
  - `STT.transcribe(audio) -> text`
  - `TTS.synthesize(text) -> audio`
- **MVP 实现**：全免费开源本地——`STT` 用 `faster-whisper`（本地推理，中文可用），`TTS` 用 `edge-tts`（免费）。
- **备选实现**：效果不满意时切换到云 API（阿里云 DashScope Paraformer + CosyVoice，或讯飞开放平台，均有免费额度），仅改实现类。
- **升级路径**：实时流式版只新增一个实现（如 Realtime 双向流），状态机与上层不变。
- **边界**：纯语音 I/O，不含任何面试逻辑。

### 5. 评估报告 `evaluator`

- **做什么**：聚合全程 Q&A + 各题评分 → 结构化报告。
- **报告内容**：总评分、维度雷达（技术深度 / 表达 / 项目匹配 / 行为面等）、逐题点评、改进建议。
- **怎么用**：`make_report(state) -> InterviewReport`，流式输出渲染。

### 6. 前端工作台 `web`（前端优势体现）

- **做什么**：完整面试体验界面。
- **能力**：语音录制（MediaRecorder）、实时转写流、对话气泡、阶段进度条、报告可视化、SSE 流式渲染。
- **边界**：通过 REST + SSE 与后端通信，不含业务逻辑。

## 数据流

1. 用户上传简历 + 粘贴 JD/公司 → `intake.build_context`
2. `planner.make_plan` 生成面试计划
3. 进入 `interviewer` 状态机循环：
   - `ask` 生成问题 → 经 `TTS` 播报给前端
   - 用户语音作答 → 前端上传音频 → `STT` 转写 → `answer`
   - `evaluate` 打分 → `route` 决定追问 / 下一题 / 换阶段 / 结束
4. 全部阶段完成 → `evaluator.make_report` → 前端渲染报告

## 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 后端 | Python + FastAPI | AI 应用工程师最高频栈 |
| Agent | LangGraph | 命中 JD 关键词，状态机可观测可回放 |
| LLM | Claude（Opus/Sonnet） | 强结构化输出与长上下文 |
| 语音 | STT/TTS adapter（默认 faster-whisper + edge-tts 免费本地，备选阿里云/讯飞云 API） | 零成本起步，接口抽象，预留升级 |
| 前端 | Next.js + TypeScript + Tailwind + shadcn/ui | 命中全栈 JD，发挥前端优势 |
| 可观测 | Langfuse | JD 关键词，trace 整条 Agent 链路 |
| 存储 | SQLite（MVP）→ Postgres | 存会话 / 报告；MVP 不引入向量库 |
| 部署 | Docker Compose | 一键启动，简历可验证 |

## 错误处理

- 简历解析失败 → 给出明确提示，允许手动粘贴纯文本兜底。
- LLM 结构化输出不合规 → schema 校验 + 一次重试，仍失败则降级为非结构化并标记。
- STT/TTS 调用失败 → adapter 层重试 + 超时；前端提示「重新录音」。
- 状态机防护：追问预算上限、单场面试最大轮次上限，杜绝无限循环与失控成本。
- Token / 成本：记录每场面试 token 消耗，Langfuse 可见。

## 测试策略

- `intake` / `planner` / `evaluator`：固定输入样例（mock 简历 + JD），断言结构化输出 schema 与关键字段。
- `interviewer` 状态机：用 mock LLM 与脚本化回答，测试路由分支（追问 / 下一题 / 换阶段 / 结束）与追问预算终止。
- `voice` adapter：接口契约测试 + mock 实现，保证可替换。
- 前端：关键交互（录音、流式渲染、报告展示）用 Playwright/Vitest（命中 JD 测试要求）。
- 端到端：一条「完整面试 → 报告」的冒烟用例。

## MVP 范围（YAGNI）

**第一版只做**：上传简历 + 粘贴 JD/公司 → 回合制语音面试（完整 6 阶段 + 追问）→ 评估报告。单用户、本地跑、无登录。

**明确推迟到 V2+**：
- 实时流式语音（架构预留，不实现）
- 公司信息自动联网调研 RAG（V2 正好补 RAG 能力格）
- 多用户 / 账号 / 数据库化
- 数字人 / 视频面试官
